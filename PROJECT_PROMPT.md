> Базовий опис проєкту `TelegramChick`.

### 1. Опис застосунку

- Назва проєкту: `TelegramChick`
- Короткий опис:
  Система для обліку товарів і партій по магазинах, контролю термінів придатності, фіксації дій співробітників і автоматичних Telegram-сповіщень.
- Основна бізнес-ціль:
  Зменшити втрати товару через пропущені терміни придатності та зробити керований процес перевірки, реакції і ескалації.
- Основна цінність:
  Показати повний операційний цикл:
  довідник товарів -> партії по магазинах -> перевірка expiry -> Telegram-нагадування -> дія співробітника -> за потреби ескалація адміністратору -> логування результату.

### 2. Основні задачі системи

1. Зберігати довідник товарів з `barcode`, `article`, назвою, категорією та одиницею виміру.
2. Зберігати партії товарів окремими записами по магазинах.
3. Дозволяти одному товару мати кілька партій у межах одного магазину з різними `expiry_date`.
4. Автоматично шукати товари, для яких наближається термін придатності.
5. Надсилати повідомлення співробітникам магазину в Telegram.
6. Дозволяти співробітнику обрати дію по позиції та залишити текстову примітку.
7. Дозволяти ескалацію до адміністратора магазину, якщо потрібне окреме рішення.
8. Зберігати історію дій: хто, що і коли змінив.
9. Накопичувати дані для подальших звітів і статистики по магазинах, товарах, партіях та працівниках.

### 3. Основні сценарії користувача

1. Система або користувач додає товар у довідник товарів.
2. Користувач додає або оновлює партію товару для конкретного магазину.
3. Система щоденно перевіряє партії, у яких наближається `expiry_date`.
4. Система надсилає Telegram-повідомлення співробітнику магазину.
5. Співробітник відкриває кейс і обирає дію по партії.
6. Співробітник залишає обов'язкову примітку.
7. Якщо рішення неочевидне, співробітник ескалує кейс адміністратору магазину.
8. Адміністратор приймає фінальне рішення і залишає свій коментар.
9. Усі зміни зберігаються в журналах дій і повідомлень.

### 4. Цільова архітектура

- Основна база даних:
  `MySQL`
- Автоматизація та бізнес-логіка:
  `n8n`
- Операційний інтерфейс для співробітників:
  `Telegram Bot`
- Додатковий інтерфейс для керування і перегляду:
  `Web app`

Розподіл відповідальності:

- `MySQL`:
  основне джерело правди для товарів, магазинів, користувачів, партій та журналів.
- `n8n`:
  sync, перевірка термінів, Telegram-інтеграція, callback-сценарії, workflow ескалацій.
- `Telegram Bot`:
  щоденна операційна робота співробітників і адміністраторів.
- `Web app`:
  зручний інтерфейс поверх тієї ж БД для керування, перегляду, аналітики і майбутніх розширень.

### 7. Ключові правила даних

- Один товар може існувати багато разів у `product_batches` в межах одного магазину.
- Базова операційна унікальність партії:
  `product_id + store_id + expiry_date`
- Якщо процес вимагатиме точнішого розділення, додатково враховувати `delivery_date` або номер поставки.
- `products.barcode` повинен бути унікальним.
- `users.user_chat_id` повинен бути унікальним.
- Усі критичні дії повинні логуватись у `activity_log`.
- Усі факти надсилання повідомлень повинні фіксуватись у `notification_log`.

### 8. Статуси та довідники

#### `check_status`

- `new`:
  нова партія без активної перевірки.
- `pending`:
  позицію потрібно перевірити після нагадування.
- `reviewed`:
  кейс відкрито, але рішення ще не завершено.
- `discussion_required`:
  потрібне рішення адміністратора.
- `completed`:
  робота по позиції завершена.
- `overdue`:
  термін придатності вже минув, а кейс не закрито.

#### `action_taken`

- `removed`
- `left_on_shelf`
- `discounted`
- `returned`
- `checked_ok`
- `not_found`
- `other`

Для `other` примітка є обов'язковою.

#### `users.role`

- `user`

- `admin`

### 9. Telegram-логіка

- Telegram є основним операційним інтерфейсом для співробітників.
- Повідомлення повинні містити:
  товар, магазин, кількість, `expiry_date`, поточний статус і варіанти дії.
- Бот повинен працювати зі звичайними повідомленнями і `callback_query` від inline-кнопок.
- Ідентифікація користувача виконується через `user_chat_id`.
- Для сценарію `discussion_required` повідомлення повинно окремо надсилатися адміністратору магазину.
- Для багатокрокових сценаріїв варто мати `user_sessions` або аналогічний механізм стану.

Базовий flow:

1. Workflow знаходить партії, у яких до `expiry_date` залишилось `notified_days`.
2. Для таких партій ставиться `check_status = pending`.
3. Співробітнику надсилається повідомлення в Telegram.
4. Співробітник обирає дію і додає примітку.
5. Дані записуються в `product_batches` і `activity_log`.
6. Якщо потрібна ескалація, кейс переходить у `discussion_required`.
7. Адміністратор приймає рішення, яке теж логуються.

### 10. Web-інтерфейс

Web не є основним операційним каналом для магазину, але потрібен для:

- перегляду довідників товарів;
- перегляду і редагування партій;
- роботи зі статусами;
- перегляду журналу дій;
- керування довідником магазинів і користувачів;
- налаштування Telegram-інтеграції;
- майбутньої аналітики та звітів.

### 11. Аналітика і звіти

Система повинна накопичувати дані для подальшого аналізу:

- кількість партій по магазинах;
- кількість `pending`, `completed`, `overdue` кейсів;
- які дії найчастіше застосовуються;
- які категорії товарів найчастіше потрапляють у перевірку;
- середній час від нагадування до закриття кейсу;
- кількість `discussion_required` кейсів;
- активність співробітників і адміністраторів;
- топ товарів і магазинів за простроченнями.

### 12. Scope поточної ітерації

- Пріоритет:
  побудувати робоче ядро процесу навколо товарів, партій, магазинів, користувачів, Telegram-сповіщень і журналів.
- Дозволені спрощення:
